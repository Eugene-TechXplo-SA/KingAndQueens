# スコープ（In / Out）

> 本仕様書では、KING and QUEEN の **Phase1（MVP相当）で成立させるべき業務・機能・状態遷移**を中心に定義する。
> Phase2で拡張予定の機能は将来方針として位置づけるが、**Phase1の完成判定（受入）には含めない**。

## 1.1 In Scope（今回対象）

### Phase1で対象とするもの（MUST）

- ユーザー登録 / ログイン（メール等）
- 口座連携申請（ブローカー選択 + MT4/MT5口座入力）
- 管理者による口座申請の承認 / 却下（理由付き）
- 取引履歴CSVのインポート（手動）
- 原資（IB報酬）入力 + 付与量計算（原資ベース：BR-08）
- 付与台帳記録（誰に・いつ・いくら・根拠バッチ）
- ダッシュボード（保有残高 / 付与履歴 / 出金履歴）
- Izakaya Wallet 連携（受取先アドレス確定・表示）
- 出金申請（**KYC必須**、最低出金額あり）
- 管理者による出金承認 / 却下（理由付き）
- 管理者の手動送金（Polygon）および TxHash 記録
- 出金ステータス管理
  `PENDING → APPROVED → PROCESSING → COMPLETED / FAILED / REJECTED / CANCELLED`
- マスター管理（ブローカー、付与アセット、配分比率、ランク、還元率、上限、価格参照設定）
- ランク制度（自動設定）
- お知らせ（CMS：作成/配信/閲覧）
- 監査ログ（管理操作ログ）
- 緊急停止（CSV取込停止 / 付与停止 / 出金停止）

> 補足（KYCの範囲）
> Phase1では **KYCの"完了/未完了ステータス"を保持**し、未完了の場合は出金申請できない（出金ゲート）ことをMUSTとする。
> KYCの本人確認フロー自体（画面/審査/外部連携）は Phase1では最小（手動運用含む）とする。

---

## 1.2 Out of Scope（今回対象外）

### Phase1では対象外（Phase2以降）

- スクレイピングによる日次自動取得
- 出金のオンチェーン自動送金
- PoR / 透明性ダッシュボード
- LINE連携
- 通知の高度化（障害アラートの外部通知、プッシュ通知等）

※Phase1の通知は **CMS（お知らせ）表示**を中心とし、外部通知（LINE/Push）は対象外とする。

## 1.3 Phase1 / Phase2 比較表

| 項目 | Phase1（MUST） | Phase2（拡張） |
|------|---------------|---------------|
| ブローカー連携 | CSV手動取込 | スクレイピング自動取得 |
| 出金送金 | 管理者が手動送金（Polygon） | オンチェーン自動送金 |
| 履歴/透明性 | ダッシュボード + 履歴表示 | PoR/透明性ダッシュボード |
| 通知 | CMS（お知らせ）中心 | LINE連携 / 高度通知 / 外部アラート |
| 運用 | 手動介入前提の安全運用 | 自動化率向上 |

## 1.4 Phase1の初期マスター設定方針

- **付与アセット**はマスター管理とし、Phase1の初期値として以下を対象とする
  `BTC, ETH, XRP, USDT, USDC, JPYR, JPYC, IZAKAYA, KQ`
- **提携FX業者**はマスター管理とし、Phase1の初期値は **添付資料（IBログインリスト）を基準**に登録する
  - ただし、要件/仕様書には **ログインID・パスワード等の機密情報は記載しない**（運用資料として別管理）
- 上記は初期設定方針であり、詳細なマスター項目定義（表示名、内部識別子、並び順、有効/無効、CSVフォーマット差分、除外条件等）は下位設計で定義する
