# ロジック定義

## 業務ルール一覧

| ルールID | ルール名 | 要約 |
|----------|----------|------|
| BR-01 | 口座連携の還元対象化 | **承認済み口座のみ**が取引集計・付与対象 |
| BR-02 | 付与成立ルール | **原資入力→CSV取込→計算→確定→台帳記録→残高反映**で成立 |
| BR-03 | 出金成立ルール | Wallet連携 + KYC + 残高/最低額 + 管理者承認 + **送金結果記録**で成立 |
| BR-04 | 監査ログ必須ルール | 重要操作は監査ログ必須 |
| BR-05 | 異常検知時の手動確認 | 上限超・重複疑い等は自動確定せず**手動確認/停止** |
| BR-06 | 付与アセット・配分ルール | 付与対象アセットと配分比率は**マスター管理**、合計100% |
| BR-07 | 提携FX業者初期設定 | 業者マスターは**IBログインリスト**を基準に初期登録 |
| BR-08 | 原資・還元率・配分の整合性 | 整合性が取れない場合は**付与確定不可** |
| BR-09 | 二重計上防止（台帳不変） | **同一バッチ/同一対象**の二重付与を禁止（台帳は追記型） |
| BR-10 | ランク制度ルール（LEV_USD） | ランクに応じて**還元率倍率**が変動。昇格は**累計獲得報酬USD（LEV_USD）**、降格は**90日非アクティブ** |
| BR-11 | キャンペーン倍率ルール | 期間限定/入会特典等で**追加倍率**を適用 |

---

## BR-01 口座連携の還元対象化

口座連携申請は、**Admin承認（APPROVED）** により還元対象口座となる。

- `APPROVED`：還元対象（取引集計・付与計算の対象）
- `REJECTED`：対象外（却下理由保持）
- `PENDING`：対象外

> 「申請済み」と「還元対象化済み」は別概念。

## BR-02 付与成立ルール

付与（= ユーザー残高に増分が反映されること）は、以下を**全て満たした場合に成立**する。

### 事前準備（マスター設定）

Adminが以下をあらかじめ設定しておく。

- 配分比率（BR-06）、還元率、参考レート（PRC-01）
- ランク設定（BR-10）、キャンペーン設定（BR-11）
- ブローカー別除外条件（BRK-01）
- 異常検知・上限（LIM-01）

### 付与フロー

1. Adminがブローカー別の **取引履歴CSVをアップロード** → 取込バッチ作成・検証
2. CSVの「アフィリエイト報酬」をトレーダー名（口座紐付け）で集計 → **原資入力欄にプリフィル**
3. Adminが原資額を **確認/変更** して確定する（BR-08）
4. 還元対象口座（BR-01）に対し、**除外条件**を適用する（BRK-01）
5. ユーザー別に **付与量を計算**（原資 × 還元率 × ランク倍率 × キャンペーン倍率 → 配分比率 → 参考レート → 端数処理）
6. **上限/異常判定**を通過する（BR-05）
7. Adminが **プレビューで確認** する
8. Adminが **付与実行を確定** する
9. 付与台帳へ記録され、**ユーザー残高へ反映**される（BR-09）
   併せて、ランク判定に用いる **USD評価額（LEV_USD用）** を台帳に固定記録する（BR-10）

> **計算完了だけでは付与は成立しない。台帳記録と残高反映まで完了して初めて付与成立**とする。

### 付与単位

- 原則「**対象月（締め期間）**」単位で確定
- 取込バッチは「**対象月 × ブローカー**」等の単位で管理し、確定後の再確定は不可（BR-09）

### 付与確定に進めない条件

- 原資額が未入力、または整合性が取れない（BR-08）
- CSVフォーマット不備／必須項目欠落
- 必須マスター不足（配分比率、還元率、参考レート未設定）
- 重複取り込み疑い
- 上限超過・急増など異常検知（BR-05）

### 取消・修正の原則（台帳不変）

- 誤付与は **取消（マイナス計上）** または **調整（delta）** を台帳に追記
- 既存行の直接書き換えはしない（監査性）

## BR-03 出金成立ルール

出金は以下を全て満たした場合に成立する。

1. ユーザーが **Izakaya Wallet連携済み**
2. ユーザーが **KYC完了**
3. **最低出金額**を満たし、**残高不足がない**
4. ユーザーが出金申請→`PENDING`
5. Admin承認→`APPROVED`
6. Admin送金開始→`PROCESSING`
7. Admin手動送金→**TxHash/送金結果を記録**
8. `COMPLETED`（成功）または `FAILED`（失敗）に更新

### 出金申請の判定表

| 条件 | 結果 |
|------|------|
| Wallet連携済み & KYC済み & 最低出金額OK & 残高OK | 申請受付可 |
| Wallet未連携 | 申請不可 |
| KYC未完了 | 申請不可 |
| 最低出金額未満 | 申請不可 |
| 残高不足 | 申請不可 |

### 重複送金防止

- `APPROVED` は完了ではなく **送金対象化**
- `COMPLETED` の根拠は **TxHash必須**
- 再送手順は下位設計で定義

## BR-04 監査ログ必須ルール

重要操作は「誰が・いつ・何を・どの対象に・なぜ・どう変えたか」を追跡可能にする。

### 記録項目

| 項目 | 内容 |
|------|------|
| 実行者 | admin_id |
| 実行日時 | timestamp |
| 操作種別 | action |
| 対象 | entity_type / entity_id |
| 差分 | 変更前 / 変更後 |
| 理由 | 自由記述 or 理由コード |

### 対象操作一覧

| カテゴリ | 操作 |
|----------|------|
| 口座連携 | 承認/却下 |
| 取引取込 | CSV取込確定（取込バッチ） |
| 付与 | 原資入力、付与確定、付与取消、台帳delta調整 |
| 配分・ランク | 配分比率変更、ランク倍率変更、**ランク閾値（LEV_USD閾値）変更（effective_from含む）**、還元率変更 |
| マスター | 還元率、上限、キャンペーン、価格参照、付与アセット設定変更 |
| 出金 | 承認/却下、TxHash記録、ステータス更新 |
| 運用 | 緊急停止ON/OFF |
| ユーザー管理 | 情報直接編集、BAN/解除、退会処理、KYC状態更新 |

## BR-05 異常検知時の手動確認

上限超過、重複、急増等の異常を検知した場合、**自動確定せず手動確認/停止対象**とする。

### 検知対象

- 同一データの重複取込疑い
- 集計ロットの急増
- 異常な付与量（上限超過）
- 異常な出金申請

### 運用方針（Phase1）

- **付与額（USD）/ 付与数量**の上限をマスター（LIM-01）で設定
- 上限を超える場合、**自動確定せず管理者の手動承認**を要求
- 内部アラート（Slack/メール等）は下位設計で定義

> MVPは安全性優先。緊急停止スイッチとセットで運用。

## BR-06 付与アセット・配分ルール

### ルール定義

- 付与対象アセットおよび配分比率は **マスター管理**（DST-01）
- 配分比率の合計は **100%**（=1.0）
- 比率変更は監査ログ対象

### Phase1 初期登録アセット（確定：9種）

`BTC` / `ETH` / `XRP` / `USDT` / `USDC` / `JPYR` / `JPYC` / `IZAKAYA` / `KQ`

### 配分の適用

ユーザー還元原資を配分比率に従って各アセットに按分し、各アセットの参考レートで数量に変換する。

## BR-07 提携FX業者初期設定

- 提携FX業者はマスター管理とし、初期登録は **IBログインリスト** を基準
- CSV取込対象は **マスター登録済み業者**に限定
- **機密情報（ID/Password等）は仕様書に記載しない**（別管理）

## BR-08 原資・還元率・配分の整合性ルール

### 付与計算モデル

### ステップ1：ユーザー別IB報酬の算出

- CSV取込時に、トレード単位の「アフィリエイト報酬」をトレーダー名（口座紐付け）で集計
- 集計結果がブローカー別の **原資入力欄にプリフィル** される
- Adminは確認の上、**変更可能**（運営裁量で調整するケースに対応）

### ステップ2：ユーザー還元原資の算出（USD基準）

```
ユーザー還元原資(USD) = ユーザー別IB報酬(USD) × 還元率 × ランク倍率（BR-10） × キャンペーン倍率（BR-11）
```

### ステップ3：アセット別付与量の算出

```
各アセット付与額(USD) = ユーザー還元原資(USD) × 配分比率（BR-06）
各アセット付与量 = floor( 各アセット付与額(USD) ÷ 参考レート(USD) × 10^token_decimals )
```

### 端数処理（D-03）

- KQトークン：**token_decimalsに従い最小単位で切り捨て**
- その他の通貨：**token_decimalsに従い最小単位で切り捨て**
- （※「1xx単位」などの粒度は下位設計で確定）

### 整合性チェック（付与確定の前提条件）

| チェック項目 | 不備時の動作 |
|-------------|-------------|
| 対象月の原資額（IB報酬）が確定済み | 付与確定不可 |
| 還元率が設定済み | 付与確定不可 |
| 配分比率の合計が100% | 付与確定不可 |
| 各アセットの参考レートが設定済み | 付与確定不可（または当該アセット除外） |
| 配分後の付与額合計がユーザー還元原資と一致 | 付与確定不可（端数許容内を除く） |

### 計算例（ユーザーA）

- ユーザー別IB報酬（CSV集計→Admin確定）: $100
- 還元率: 20%、ランク倍率: 1.0、キャンペーン倍率: 1.5（例: 春キャンペーン）
- ユーザー還元原資: $100 × 20% × 1.0 × 1.5 = **$30**
- 配分：BTC 40% / USDT 30% / ETH 20% / KQ 10%
  - BTC $12 / USDT $9 / ETH $6 / KQ $3（合計 $30）

> **決定済み（TBD-01）**: 付与計算モデルは **原資ベース（USD）** を採用。CSVの「アフィリエイト報酬」列からユーザー別IB報酬を集計し、原資入力欄にプリフィル→Admin確定する。ロット数はCSVに含まれるが監査/参照用であり、計算には使用しない。

## BR-09 二重計上防止（台帳不変）

### ルール定義（最重要）

- 同一取込バッチに対して **付与確定は1回のみ**
- 台帳は不変（追記型）とし、修正は **取消/調整** で表現
- 同一取引が **重複計上されない**こと

### 台帳に必須のキー

| キー | 用途 |
|------|------|
| `batch_id` | 取込バッチID |
| `target_month` | 対象月 |
| `account_id` | 対象口座ID |
| `entry_type` | `GRANT / CANCEL / ADJUST` |
| `asset_code` | BTC/USDT/JPYR/KQ… |
| `dedupe_key` | 冪等キー（例：`batch_id + user_id + entry_type + asset_code`） |

### 台帳に記録するLEV項目（BR-10用）

| 項目 | 内容 |
|------|------|
| `rate_usd_at_grant` | 付与確定時点の参考USDレート |
| `value_usd_at_grant` | `amount × rate_usd_at_grant` |
| `signed_value_usd_at_grant` | 取消/調整はマイナスで記録 |

### 運用

- 付与確定済みバッチへの再確定は不可
- 取消は「マイナス付与」として台帳に追記し残高を相殺
- 調整（delta）も台帳に追記し、理由を必須とする
- **LEV_USDも同様に符号付きで相殺/調整される**

## BR-10 ランク制度ルール（LEV_USD / 月次判定）

### ルール定義

ユーザーにはランクが付与され、ランクに応じて **還元率倍率** が変動する。
昇格判定は **累計獲得報酬USD（LEV_USD）** を用いる。
降格は **90日取引なし**（非アクティブ）時のみ行う。閾値変更による即時降格は行わない。

### 用語（ランク母数）

- **LEV_USD**：Lifetime Earned Value in USD
  `LEV_USD = Σ signed_value_usd_at_grant`
  ※付与確定時点のUSD評価額を台帳に固定保存し、価格変動で再評価しない
- `last_trade_at`：CSV取込により更新される最終取引日時

### ランク構造（マスター管理：RNK-01）

| 項目 | 内容 |
|------|------|
| ランク名 | Member / Silver / Gold（Phase1） |
| 還元率倍率 | 例：1.0 / 1.5 / 2.0 |
| **LEV_USD閾値** | Silver $X / Gold $Y 等 |
| **effective_from** | 適用開始日（将来適用のみ） |
| version | 変更履歴のため保持 |

### 昇格条件（自動）

- 月次判定時点で、`LEV_USD` が閾値以上なら到達可能な最高ランクに更新

### 降格条件（自動）

- 月次判定時点で `last_trade_at` が **90日以上更新なし**の場合、**1段階降格**
  （Gold→Silver→Member）

### 判定タイミング（Phase1）

- **月次バッチ**で実施（例：毎月1日）
- デフォルトは Member（倍率1.0）

### 閾値マスター変更ルール（重要）

- 閾値は運営がマスター画面で変更可能
- 変更は `effective_from` を持ち **将来適用のみ（遡及しない）**
- 変更は **監査ログ必須（理由必須）**

## BR-11 キャンペーン倍率ルール

### ルール定義

キャンペーンにより、対象ユーザーの付与計算時に **追加倍率** を適用する。

### キャンペーン種別

| 種別 | 例 | 対象 |
|------|---|------|
| 期間限定（全体） | 春キャンペーン：期間中全ユーザーに1.5倍 | 全対象ユーザー |
| 入会特典（個別） | 入会から1ヶ月間はxx%UP | 条件を満たすユーザー個別 |

### キャンペーンマスター（CMP-01）

| 項目 | 内容 |
|------|------|
| キャンペーン名 | マスターで定義 |
| 倍率 | 例：1.5 |
| 期間 | 開始日 / 終了日 |
| 対象条件 | 全体 or 個別 |

### 適用ルール

- 付与計算時に対象ユーザーの **有効な倍率** を適用（BR-08）
- 未設定は 1.0
- 複数重複時の合算/最大値/優先度は下位設計で定義
- 作成/変更は監査ログ対象

## Off-System 運用フロー

Adminがシステム外で実施し、結果のみをシステムに記録する作業。

| # | 作業 | 実施場所 | システム記録 |
|---|------|----------|-------------|
| 0 | マスター設定（配分比率/還元率/レート/ランク倍率/ランク閾値/キャンペーン/上限） | システム内 | 監査ログ必須 |
| 1 | ブローカー管理画面ログイン | システム外 | - |
| 2 | 取引履歴CSV取得 | システム外 | - |
| 3 | CSVアップロード → 原資プリフィル → Admin確認/変更 | システム内 | 監査ログ必須 |
| 4 | 付与計算 → プレビュー確認 → 付与確定 | システム内 | 監査ログ必須 |
| 4.5 | **月次ランク判定（LEV_USD + 90日非アクティブ）** | システム内 | 監査ログ必須 |
| 5 | 付与用アセット調達（KQ購入含む） | システム外（CEX/DEX） | - |
| 6 | 出金の実送金（Polygon） | システム外 | - |
| 7 | TxHash記録・出金状態更新 | システム内 | 監査ログ必須 |
