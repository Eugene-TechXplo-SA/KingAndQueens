## 2. 共通データ形式

### 2.1 ID形式

- 主要リソースIDは **UUID** を使用する（例：`user_id`, `withdrawal_request_id` など）。

### 2.2 日時形式

- 日時は **ISO 8601** を使用（例：`2026-02-27T12:34:56Z`）。
- タイムゾーンは原則 **UTC**（表示はクライアント側でローカライズ）。

### 2.3 数値・金額・トークン数量

- トークン数量（KQ）は **浮動小数で保持しない**。
- 受け渡しは **最小単位の整数**を基本とし、`_atomic` を使用する（例：`amount_atomic`, `fee_atomic`）。
- `token_decimals`（最小単位の桁数）は システム設定として固定（TBD）。
- 入力が小数の場合、最小単位に変換できない値は **拒否**（丸めない）。

### 2.4 文字コード（JSON/CSV）

- JSONはUTF-8。
- CSVインポートも原則UTF-8（BOM有無はTBD）。
- 改行は `LF` 推奨（`CRLF` 許容はTBD）。

## 3. 共通エラーレスポンス

### 3.1 エラースキーマ

エラー時は以下を返す（フィールド名は共通化）。

- `code`：機械判定用コード（例：`KYC_NOT_APPROVED`）
- `message`：人間向けメッセージ
- `details`：任意（配列/オブジェクト）

### 3.2 Validationエラー

- 入力不正は `details` にフィールド単位の理由を含める。
- 例：`details = [{ field: "amount_atomic", reason: "must be > 0" }]`

### 3.3 HTTPステータスの基本方針

| コード | 意味 |
|--------|------|
| `401` | 未認証 |
| `403` | 権限なし / 前提不足（例：KYC未承認）※`409`に寄せる場合は全体で統一 |
| `404` | 存在しないリソース |
| `409` | 状態不正 / 競合 / 重複（二重実行、ユニーク違反など） |
| `422` | 入力値不正 |
| `429` | レート制限超過（採用する場合） |
| `5xx` | サーバー側障害 |
| `503` | 停止スイッチ等で一時不可 |

## 4. 認証・認可

### 4.1 認証方式（概要）

- 認証方式は JWT（または同等トークン）を前提（TBD）。
- ヘッダ：`Authorization: Bearer <token>`

### 4.2 認可（権限の原則）

- **User**：自分に紐づくリソースのみ参照/操作可能。
- **Admin**：管理対象の参照/操作可能（承認、CSV取込、付与確定、設定変更、BAN等）。

### 4.3 トークン期限・更新

## 5. 冪等性・リトライ

### 5.1 冪等性（Idempotency）

- 作成系・確定系は冪等性を考慮する。
- 冪等キーは以下のいずれか（APIごとに採用を明記）。
  - `Idempotency-Key`（ヘッダ推奨）
  - `client_request_id`（ボディ）
- 同一キーの重複リクエストは **同一結果を返し、重複作成しない**。

### 5.2 リトライ方針

- リトライ可：`429`、`5xx`
- リトライ不可が多い：`409/422`（入力・状態の修正が必要）
- バックオフは指数バックオフ推奨（具体値TBD）

## 7. トレーシング（Request ID）

- クライアントは `X-Request-Id` を付与推奨（UUID）。
- サーバーはログに記録し、レスポンスにも同値を返す。

---

## 8. ページング・ソート・フィルタ（標準）

### 8.1 ページング（オフセット方式）

- `page`（1始まり）
- `page_size`（上限TBD、例：100）
- レスポンスは `items` と `page_info`（`page`, `page_size`, `total`）を返す。

### 8.2 ソート

- `sort`：`field` で降順、`field` で昇順
- 例：`sort=-requested_at`

### 8.3 フィルタ

- `status`（該当一覧）
- 期間：`from` / `to`（ISO8601, UTC）
